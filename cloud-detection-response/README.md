# Cloud Detection and Response

## What this folder is

A practitioner's reference for detection and incident response in cloud environments — log architecture, native detection services, threat hunting, and IR runbooks for cloud-native incident classes on AWS, Azure, and GCP. The material here is what I put in front of a security team when the question is: *we have CloudTrail / Activity Logs / Audit Logs streaming somewhere, GuardDuty / Defender / SCC turned on, and a SIEM that nobody trusts — what is the detection-and-response architecture we should actually be running?*

## The organizing principle

Detection-and-response in cloud environments inverts the order most security teams are used to. In on-premises environments, log collection is the hardest problem and detection logic is the secondary concern. In cloud environments, log collection is mostly solved (every API call is logged by default; the question is where you store it and what you do with it), and detection logic is the hard problem. The leverage shifts accordingly: an excellent log pipeline that feeds mediocre detections produces alert fatigue and slow IR; a mediocre log pipeline that feeds excellent detections produces the inverse problem (good detections that you cannot investigate). The patterns here treat the pipeline and the detection logic as a single design problem with two halves.

The folder is opinionated about three things. First, that **the native detection services are good enough to be the foundation, and not good enough to be the whole solution** — GuardDuty / Defender for Cloud / Security Command Center catch the obvious classes (cryptomining, leaked credential, known-bad indicator), and they miss the targeted classes (lateral movement through legitimate APIs, slow-rate reconnaissance, supply-chain compromise via a trusted role). The right architecture uses the native services for breadth and adds custom detections for depth. Second, that **CloudTrail / Activity Log / Audit Log enrichment is the highest-leverage detection-engineering work in cloud** — joining an API call with the identity that made it, the resource it touched, the network it came from, and the prior pattern of behavior for that identity is what turns a raw log into a detection signal. Third, that **incident response in cloud has a different shape than IR on-prem** — the containment options are different (isolate-via-VPC vs disconnect-the-cable), the forensics options are different (instance snapshot vs disk image), and the eradication options are different (revoke-the-session-token vs reimage). The runbooks reflect that.

## How to use this section

**If you are designing a detection-and-response architecture from scratch**, start with `log-architecture.md` and `siem-integration.md`. The pipeline is the foundation; detection logic is what you build on top.

**If you are responding to a live incident**, the runbook documents are the destination. The runbook structure is consistent across the folder (scope, detection, containment, forensics, eradication, recovery, lessons learned) so a security engineer who has read one can navigate any of the others under time pressure.

**If you are tuning native detections for noise**, `guardduty-defender-scc.md` includes the noise-to-signal workflow. Most teams adopt a "rule of three" before suppressing a finding class; the document describes when that rule is appropriate and when it is the wrong move.

## What this section is not

- **A complete SIEM tutorial.** Splunk / Sentinel / Chronicle / Elastic each deserve their own documentation and the vendors maintain it better than this folder could.
- **A general-purpose IR guide.** The runbooks target cloud-native incident classes specifically. Endpoint compromise, ransomware on a corporate laptop, and traditional network intrusion are out of scope.
- **A threat-intelligence reference.** TI feed selection, indicator management, and intel-product consumption are adjacent to this folder but not within it.
